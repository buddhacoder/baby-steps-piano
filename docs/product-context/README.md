# Baby Steps Product Context

This folder holds durable source context for the Baby Steps product thesis. These files are not the always-loaded coding instructions; they are reference material for agents or humans who need to understand why the app exists, how its pedagogy is framed, and how claims should be bounded.

For everyday coding work, read `context/` first. Use this folder when a task touches product positioning, pedagogy, learner value, study design, launch content, claims language, or the overall app direction.

## Documents

- `Baby_Steps_White_Paper_v1.0.pdf` - Evidence-informed framework and value thesis. Use this for the full product/pedagogy rationale, evidence ladder, accessibility commitments, telemetry implications, and claim-language boundaries.
- `Baby_Steps_Design_Based_Pedagogy_Paper.pdf` - Practitioner-facing model explanation. Use this for the Touch, Move, Feel, Notice, Name, Transfer sequence; Baby Step criteria; Practiced, Confident, Mastered rubric; and the Left-Hand Chord Freedom path.
- `Baby_Steps_Study_Protocol.pdf` - Validation roadmap. Use this for study phases, hypotheses, outcomes, eligibility, ethics, accessibility, reporting, and the rule that comparative claims require completed comparative evidence.
- `content-manager.md` - Adapter note for content-production workflows tied to Baby Steps.

## Distilled Product Thesis

- Baby Steps is for adult beginner and re-entering piano learners, especially people who need low-overload, ADHD-aware, emotionally safe re-entry into music learning.
- The core learning unit is a tiny playable musical action, not a lecture topic. A Baby Step should be small enough to start, meaningful enough to hear, reversible, theory-deferred, and transferable.
- The app should sequence experience before language: touch, move, feel, notice, name, then transfer. Theory still matters; it lands after the learner has a sound and movement to attach it to.
- Progress should avoid pass/fail harshness. Use Practiced, Confident, and Mastered states to separate first contact, in-session reliability, delayed retrieval, and transfer.
- The flagship path is Left-Hand Chord Freedom: make the left hand feel like a reliable harmonic anchor so adult learners can experience musical freedom sooner.
- Claims must stay honest. Until validation phases are completed, describe the model as evidence-informed and testable, not proven, clinical, therapeutic, or superior.
- The app should behave like a guided thinking/practice partner, not a static course catalog: model, attempt, immediate feedback, self-check, saved state, return-after-lapse support, and transfer prompts.
- Accessibility and nervous-system-safe design are product requirements, not decoration: reduced overload, clear re-entry, low-threat retries, readable controls, and user control over motion/audio/haptic intensity.

## Agent Guidance

When implementing features, do not load the PDFs by default. Instead:

1. Read the six files in `context/`.
2. Use this README for fast product alignment.
3. Open the PDFs only when the task depends on pedagogy, research framing, claims, study design, or product positioning.
4. Keep any product claim aligned with the validation stage described in the white paper and protocol.
